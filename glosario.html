3  Glosario de Términos (Flutter & Dart)

Glosario técnico con referencias cruzadas a las unidades donde se cubre cada tema en profundidad.

3.1 A

  • AAB (Android App Bundle): Formato de publicación de Google Play que optimiza el tamaño de la app para cada dispositivo. → Ver: Unidad 12: Build y Distribución
  • AdMob: Plataforma de monetización de Google para apps móviles. → Relacionado: Unidad 12: Build y Distribución
  • API (Application Programming Interface): Contrato que define cómo dos sistemas se comunican. → Ver: Unidad 7: Integración con APIs

3.2 B

  • Bloc (Business Logic Component): Patrón de gestión de estado basado en streams. → Relacionado: Unidad 5: Gestión de Estado
  • Build Context: Objeto que contiene información sobre la ubicación del widget en el árbol de widgets.

3.3 C

  • Clean Architecture: Patrón de diseño que separa el código en capas (Domain, Infrastructure, Presentation) para mejorar la mantenibilidad. → Ver: Unidad 6: Clean Architecture
  • ConsumerWidget: Widget de Riverpod que puede escuchar cambios en el estado sin necesidad de un StatefulWidget. → Ver: Unidad 5: Riverpod
  • Cross-References: Referencias entre secciones del libro. Ejemplo: La capa DOMAIN (→ Chapter 15) es independiente de Flutter.

3.4 D

  • Dart: Lenguaje de programación optimizado para UI, desarrollado por Google.
  • DDD (Domain Driven Design): Metodología de diseño de software centrada en el dominio del negocio. → Ver: Unidad 6: DDD en Flutter
  • Dependency Injection: Patrón para inyectar dependencias desde fuera, no crearlas dentro. → Ver: Unidad 6: Clean Architecture
  • Dio: Paquete potente para realizar peticiones HTTP con soporte para interceptores. → Ver: Unidad 7: APIs HTTP
  • Drift: Biblioteca de persistencia (SQLite) para Dart y Flutter con tipado seguro. → Ver: Unidad 8: SQLite con Drift

3.5 E

  • Entity: Objeto del dominio que representa una entidad de negocio con identidad única. → Ver: Unidad 6: Clean Architecture
  • Entorno de Desarrollo (IDE): Android Studio, VS Code o IntelliJ con plugins de Flutter.

3.6 F

  • Firebase Cloud Messaging (FCM): Servicio de notificaciones push de Google. → Ver: Unidad 9: Firebase Notifications
  • FVM (Flutter Version Manager): Gestor de versiones de Flutter para proyectos con diferentes versiones.
  • Future: Objeto que representa un valor que estará disponible en el futuro (operación asíncrona).

3.7 G

  • GetIt: Biblioteca de inyección de dependencias para Dart y Flutter.

3.8 H

  • Hot Reload: Característica de Flutter que permite inyectar cambios de código directamente en la app sin reiniciarla. → Ver: Unidad 3: Primeros Pasos
  • Http Client: Componente para realizar peticiones de red. → Ver: Unidad 7: Integración con APIs

3.9 I

  • InheritedWidget: Widget base para compartir estado hacia abajo en el árbol de widgets.
  • Isolate: Concurrencia en Dart - procesos separados con memoria independiente.

3.10 L

  • Layout: Estructura visual de la interfaz. → Ver: Unidad 4: Layouts
  • Listener: Función que se ejecuta cuando cambia el estado.

3.11 M

  • Mappers: Funciones que transforman datos entre capas (DTO → Entity, Entity → Model). → Ver: Unidad 7: Mappers y Models
  • Material Design: Sistema de diseño de Google para interfaces consistente.

3.12 P

  • Package: Librería reutilizable de código Dart/Flutter.
  • pubspec.yaml: Archivo de configuración de dependencias de Flutter.
  • Provider: Paquete oficial de gestión de estado de Flutter. → Ver: Unidad 5: Gestión de Estado
  • Pub.dev: Repositorio oficial de packages Dart y Flutter.
  • Push Notification: Notificación enviada al dispositivo aunque la app esté cerrada. → Ver: Unidad 9: Firebase Notifications

3.13 R

  • Ref (Reference): En Riverpod, objeto para leer estado sin rebuild automático.
  • Repository Pattern: Abstracción que hiding la fuente de datos real.
  • Riverpod: Gestor de estado reactivo y seguro que mejora los conceptos de Provider. → Ver: Unidad 5: Riverpod

3.14 S

  • SetState: Método en StatefulWidget para rebuilds locales.
  • SQLite: Base de datos local embebida. → Ver: Unidad 8: SQLite con Drift
  • StatefulWidget: Widget con estado mutable que puede cambiar durante su lifetime.
  • StatelessWidget: Widget inmutable que solo se construye una vez.
  • Stream: Secuencia asynchronous de eventos o valores.

3.15 T

  • TDD (Test-Driven Development): Metodología de desarrollo donde se escribe el test primero. → Ver: Unidad 14: Testing y CI/CD

3.16 U

  • useProvider: Hook de Riverpod para acceder a providers en widgets.

3.17 V

  • VoidCallback: Función sin返回值 (equivalente a Action en otros frameworks).

3.18 W

  • Widget: La unidad básica de construcción de interfaces en Flutter. Todo en Flutter es un widget. → Ver: Unidad 4: Widgets Básicos
  • Widget Tree: Árbol jerárquico de todos los widgets de la app.

3.20 📇 Flashcards de Repaso

Las siguientes flashcards te ayudan a repasar los conceptos clave del curso. Usa estas preguntas para autoevaluarte antes de los exámenes.

  • Clean Architecture

3.20.1 Pregunta 1

¿Cuáles son las 3 capas principales de Clean Architecture?

Tip

Respuesta: 1. DOMAIN - Lógica de negocio pura (Dart sin dependencias de Flutter) 2. INFRASTRUCTURE - Implementaciones concretas (APIs, Bases de datos) 3. PRESENTATION - UI y Widgets de Flutter

3.20.2 Pregunta 2

¿Por qué el DOMAIN debe ser independiente de Flutter?

Tip

Respuesta: Para mantener la portabilidad. El dominio puede usarse en cualquier plataforma (API REST, CLI, Flutter Web) sin cambios.

3.20.3 Pregunta 3

¿Qué es un NotifierProvider en Riverpod?

Tip

Respuesta: Un provider que permite mutable state (estado mutable) con métodos para modificar el estado. Ideal para CRUD operations.

3.20.4 Pregunta 4

¿Cuál es la diferencia entre FutureProvider y StreamProvider?

Tip

Respuesta: - FutureProvider: Para operaciones que devuelven un solo valor (una petición HTTP) - StreamProvider: Para flujos de datos continuos que se actualizan en tiempo real

3.20.5 Pregunta 5

¿Qué es el patrón Repository en Clean Architecture?

Tip

Respuesta: Una abstracción que define cómo se acceden los datos, separando la interfaz (contrato) de la implementación concreta (Drift, Firebase, API).

3.20.6 Pregunta 6

¿Cuál es la diferencia principal entre Firebase y Supabase?

Tip

Respuesta: - Firebase: Ecosistema cerrado de Google, propiedad - Supabase: Código abierto (open-source), basado en PostgreSQL, más control sobre los datos

3.20.7 Pregunta 7

¿Qué es un RepaintBoundary y cuándo usarlo?

Tip

Respuesta: Un widget que aísla su hijo del resto del árbol de renderizado. Usarlo cuando tienes animaciones o widgets complejos que no necesitan ser redibujados cuando cambia el estado padre.

3.20.8 Pregunta 8

¿Por qué const es importante en Flutter?

Tip

Respuesta: El compilador puede cachear widgets const porque nunca cambian, reduciendo el número de rebuilds necesarios.
